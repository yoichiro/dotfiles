# Design Doc テンプレート

Design Doc で使う**固定の章立て**と、各セクションの書き方を示す。セクションを増やしたり順序を変えたりしない。

## 全体骨格

```markdown
# <機能名>

**日付:** <YYYY-MM-DD>

## 概要
（...）

## 要件
- （...）
- （...）

## 設計

### 定数
- （...）

### 型定義
（TypeScript / 対応言語のコードブロック）

### アーキテクチャ
（...）

### Firebase Firestore のコレクション/ドキュメントスキーマ
（...）

### Firebase Storage
（...）
```

---

## セクション別の書き方

### `# <機能名>`

機能を端的に表す日本語タイトル。末尾に「取り込み」「同期」「認証」「配信」「集計」など、動作を表す名詞を付けるとスコープが明確になる。

- 良い例: `ユーザー招待メール送信` / `売上データ日次集計` / `外部 API レートリミット制御`
- 悪い例: `メール` / `集計機能`（何をする機能か不明）

### `**日付:** <YYYY-MM-DD>`

今日の日付を ISO 8601 形式で記載する。

### `## 概要`

**目的**: 実装者が「この機能は何をするものか、結果として何がシステムに起きるか」を把握できる。

**書き方**:
- 1〜2 段落で導入を書く（何を入力として何を出力するか、どのサービスにどう保存／送出するか）
- 「本機能が実行されることで、以下が達成される。」と続けて、箇条書きで 5〜8 個の達成項目を列挙する

**テンプレート**:
```markdown
## 概要

<機能の一言説明。入力・処理の種類・出力先を含める。>

本機能が実行されることで、以下が達成される。

- <達成項目 1>
- <達成項目 2>
- ...
```

### `## 要件`

**目的**: 処理の流れをステップとして把握できる。

**書き方**:
- 箇条書きで、処理の順序どおりに記述する
- 「〜する」で終わる文に揃える
- 条件分岐（エラー処理・上限超過・キャンセル・リトライなど）も箇条書きに含める
- ネスト（インデント）を使って「各レコードに対して以下を順次実行する」の配下に詳細を書くとよい

### `### 定数`

**目的**: 実装で使う定数の値と、**それをその値にした理由**がわかる。

**書き方**:
- 箇条書きで `定数名 = 値 (ファイルパス)` の形式
- 各定数に `**理由**: ...` を添える
- 定数の値だけ書いて理由を書かないのは禁止

**例**:
```markdown
- `MAX_BATCH_SIZE = 500` (`src/services/writer.ts`)
  - 1 バッチで書き込むレコードの上限。
  - **理由**: 利用 DB のバルク API が 1 リクエスト 500 件を超えるとタイムアウトすることが観測されているため。これより大きな値を取らないこと。
```

### `### 型定義`

**目的**: 実装者が型を書く際に、フィールドと型を迷わず決められる。

**書き方**:
- プロジェクトで用いる言語のコードブロック（TypeScript の場合は ` ```typescript ... ``` `）で型を列挙する
- 各型の上にコメントで目的を書く
- 型定義ブロックの後に、`**補足（設計上の選択）:**` セクションを置き、以下のような理由を箇条書きで補足する
  - なぜ union 型や discriminated union を使ったか
  - なぜ `Date` ではなく `string` なのか
  - なぜ optional なのか
  - 永続化時にシリアライズの都合がある場合はその制約

### `### アーキテクチャ`

**目的**: 実装者がどのサービス、どのライブラリ、どのランタイムを使うべきかを**一意に決定できる**。

**サブセクション構成**:

1. **実行環境**
   - 言語／ランタイム（例: TypeScript + Node.js、Python、Go）
   - 使用するフレームワーク（HTTP／ワーカー／バッチなど）
   - コンテナ化の有無（Dockerfile があれば要約する）
   - メモリや同時実行数などのランタイムパラメータがあれば、その値と理由
   - 想定デプロイ先（例: Google Cloud Run、AWS Lambda、Kubernetes。選定理由を添える）

2. **依存する外部サービス／ライブラリ**
   - 表形式で「役割 | サービス／ライブラリ | 備考」を列挙する
   - 表の直後に「**なぜ X なのか**」という形で主要サービスの選定理由を 2〜4 個書く
   - 実装者が別の選択肢に流れないように、「別の選択肢を採らないこと」の強い言い回しを入れる

3. **モジュール構成**
   - 関係するソースファイルと責務を箇条書き
   - 「X のビジネスロジックは Y モジュールに閉じ込める」のような責務の境界を明記する

4. **重要な設計方針**
   - 5〜8 個の設計方針を番号付きリストで並べる
   - 各方針に「**理由**: ...」を添える
   - 例: 同期 vs 非同期の選択、順次処理 vs 並列処理、ストリーミング vs 一括処理、バージョニング方式、致命的エラーと回復可能エラーの判定、トランザクション境界、冪等性の確保、進捗の粒度

5. **認証と認可**
   - 認証方式（OAuth、API Key、サービスアカウント、mTLS 等）と選定理由
   - 必要なスコープ・権限の一覧
   - 秘密情報の注入方法（環境変数名、Secret Manager のキー名など）
   - トークンのライフサイクルとリフレッシュ方式（該当する場合）

### `### Firebase Firestore のコレクション/ドキュメントスキーマ`

**目的**: Firestore のどこに何を保存するかが一意に決まる。

対象機能が Firestore を利用しない場合は、以下の 1 行のみ記載する:

```markdown
本機能では利用しない。
```

利用する場合の書き方:
- 関係するコレクションを `#### <コレクションパス>` の H4 で列挙する
- 各コレクション配下に**フィールド表**を置く:

  ```markdown
  | フィールド | 型 | 説明 |
  | --- | --- | --- |
  | `uid` | string | ユーザー ID |
  ```

- 表の直後に `**設計理由:**` セクションを置き、なぜこのスキーマにしたかを 2〜4 点書く
  - なぜこのパスに置くか
  - なぜフィールドを JSON 文字列化するか（該当する場合）
  - なぜメタデータを非正規化するか（該当する場合）
  - サブコレクションの扱い、削除時の注意事項、インデックス要件

### `### Firebase Storage`

**目的**: Storage のバケットとパス規約が一意に決まる。

対象機能が Firebase Storage を利用しない場合は、以下の 1 行のみ記載する:

```markdown
本機能では利用しない。
```

利用する場合の書き方:
- 以下の項目を箇条書きで列挙し、**各項目に「なぜそうするか」の理由を併記する**

  - **バケット名**: 値と、定数が定義されているファイルパス。「実装者はバケット名を文字列リテラルで散らさず、必ずこの定数経由で参照すること」のような指示を入れる
  - **オブジェクトパス**: パス規約（例: `{uid}/{projectId}/{itemId}`）
    - **理由**: プレフィックス階層による権限制御、Firestore ドキュメントとの対応関係など
  - **コンテンツタイプ**
  - **格納内容**: 何が入っているか
    - **「なぜ X を付加するか」** のように記述する
  - **書き込みタイミング**
    - **「X より前に書くのはなぜか」** の理由を添える
  - **削除タイミング**
    - 正常終了時／キャンセル時／再同期時の挙動
    - **「旧オブジェクトを消さない／消す理由」** を明記する
