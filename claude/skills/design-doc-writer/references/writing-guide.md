# 執筆ガイド

このガイドは `design-doc-writer` スキルで Design Doc を書く際の詳細指針をまとめる。SKILL.md の執筆原則を実践する際に参照する。

## 執筆原則の詳説

### 原則 1: 実装者が手段に迷わないレベルの具体性

Design Doc は「この機能はこうあってほしい」を伝えるだけでなく、**どう作るか**までを伝える。

**目指す水準**: ドキュメントを渡された実装者が、ライブラリ選定・ステップ順序・データ形式・ファイルパスで「選択の余地に直面しない」こと。

**表現のコツ**:

- 動作 → 断定的に「〜する」「〜のこと」
- 禁止 → 「〜を採らないこと」「〜しないこと」
- 責務 → 「X は Y に閉じ込める」「Z を横断させない」

**Good**:
> 永続化層の生 SDK にはコントローラから直接触れず、リポジトリモジュール経由で操作すること。

**Bad**:
> DB にアクセスする方法はいくつかある。必要に応じて使い分けてよい。

### 原則 2: 「なぜその方式にしたか」の意思決定理由

実装者は「書いてある通り作ればいいが、なぜそうなのか」を理解していないと、例外ケースでの判断ができない。理由を添えることで、想定外の状況でも整合した判断ができる。

**配置の目安**: 定数 1 件ごと、型定義のグルーピングごと、設計方針 1 件ごとに必ず 1 つは理由を書く。Design Doc 全体で**最低でも 10 箇所以上**は「**理由:**」「**設計理由:**」「**なぜ X なのか**」などのラベル付きで理由が登場することを目標にする。

**理由の書き方**:

- 他の選択肢と比較する（例: 「X ではなく Y を選ぶのは Z が理由」）
- 制約から導く（例: 「ランタイムのメモリ上限下で動かすため」）
- 外部要因を示す（例: 「使用 API の 1 リクエスト上限が 100 件のため」）
- トレードオフを明言する（例: 「書き込み時の冗長は読み出し時のシンプルさで相殺する」）

**Good**:
> - `PAGE_SIZE = 100`
>   - **理由**: 利用している API の 1 回の取得上限が 100 件であるため。これより大きな値を指定してもサーバ側で 100 に丸められる。

**Bad**:
> - `PAGE_SIZE = 100`
>   - ページングサイズ。

### 原則 3: 日本語で記述、ただしコード識別子は英語

地の文は日本語で書くが、以下は英語のままにする。

- 定数名、関数名、クラス名、型名、変数名
- ファイルパス、ディレクトリ名
- HTTP メソッド、ステータスコード
- ライブラリ名、サービス名、モデル名、プロダクト名
- 環境変数名

日本語と英語の境界には半角スペースを入れると読みやすい（例: `API の 1 リクエスト上限`）。

### 原則 4: コードに書いてある事実を元にする

定数・関数・型は実コードから抽出する。値の出典を括弧書きで添えると、ドキュメントのメンテナンス性が上がる。

**Good**:
```markdown
- `MAX_RETRIES = 3` (`src/lib/retry.ts`)
```

**Bad**:
```markdown
- リトライ上限: 3 回くらい
```

### 原則 5: 推測を混ぜない

実装・デプロイ設定・運用方針から読み取れない事項は、断定ではなく「想定」として書くか、あるいは書かない。

**Good**:
> 想定デプロイ先: Google Cloud Run（ステートレスな HTTP サービス、オートスケーリング、ゼロスケール、統合ロギングとの親和性から選定）。

**Bad**:
> Cloud Run にデプロイされている。（← 本当にそうかコードから確認できない場合）

## よくある誤り

1. **理由を省略する**: 「軽量モデルを選ぶ」とだけ書いてしまう。→ 「コストが重量級モデルの約 1/10 で、要約用途には十分な品質が得られる」まで書く。
2. **章立てを変える**: 「今回は Firebase Storage を使わないから消す」としない。→ 使わない機能でも章は残し、「本機能では利用しない。」と 1 行書いて終える、**または** ユーザーに章立てを変えてよいか確認する。
3. **処理の流れを設計節に置く**: 要件節と重複する「ステップ 1: ...」の小節を設計節に作らない。→ 「処理の流れ」は要件節に寄せ、設計節は「定数／型／アーキテクチャ／データモデル」に集中する。
4. **抽象すぎる設計理由**: 「拡張性のため」「保守性のため」。→ 何から何へ拡張するのか、どんな保守負担が減るのかを具体化する。
5. **コードの丸写し**: 関数本体を Design Doc にベタ貼りする。→ 型定義は貼ってよいが、関数実装は貼らず責務を文章で述べる。
6. **外部サービスの調査手がかりを添えない**: 「X サービスに依存する」とだけ書く。→ ライブラリ名、公式ドキュメント URL、エンドポイントなど、実装者が最短で調査に着手できる手がかりを添える。
7. **コードベース固有の前提を埋め込みすぎる**: 具体例を出すのは良いが、特定プロジェクトにしか通じない用語を注釈なしで多用しない。新規メンバーがそのドキュメントだけ読んで理解できるかを目安にする。

## セルフチェックリスト

書き上げた後、以下を自己点検する。

- [ ] 章立てがテンプレート通りである（増減なし、順序通り。非該当なら「本機能では利用しない。」で明示）
- [ ] タイトルが「〜取り込み」「〜同期」「〜配信」など動作名詞で終わっている
- [ ] 日付が `YYYY-MM-DD` 形式で今日の日付になっている
- [ ] 「概要」に達成項目の箇条書きがある
- [ ] 「要件」が処理順序に沿った箇条書きになっている
- [ ] 「定数」の各項目に **理由** がある
- [ ] 「型定義」のブロック後に **補足（設計上の選択）** がある
- [ ] 「アーキテクチャ」に 5 つのサブセクション（実行環境／依存サービス／モジュール構成／設計方針／認証認可）がある
- [ ] 「アーキテクチャ」の依存サービス・ライブラリが表形式で示されている
- [ ] 「Firebase Firestore」の各コレクションにフィールド表と **設計理由** がある（利用する場合）
- [ ] 「Firebase Storage」の各項目に理由が併記されている（利用する場合）
- [ ] ドキュメント全体で「**理由:**」「**設計理由:**」「**なぜ X**」等の理由ラベルが 10 箇所以上ある
- [ ] ライブラリ・サービス・定数値・ファイルパスを実コードから抽出している（推測が混ざっていない）
- [ ] 「〜することもできる」「〜してもよい」のような曖昧表現を使っていない
- [ ] コード識別子は英語、地の文は日本語になっている
- [ ] プロジェクト外の読者が読んでも、暗黙の前提でつまずかない

## 調査時の tips

- 対象機能のエントリポイントは、エンドポイント定義（ルーティング／ハンドラ）や CLI コマンド定義、スケジュールトリガー設定を起点に辿る
- 大きなオーケストレーションコード（複数分岐のあるコントローラ）は、該当ブランチを Grep やシンボル検索で特定してから範囲を絞って読む
- プロジェクト概要の `README.md`・`CLAUDE.md`・`docs/` を最初に読むと、全体構造と用語を把握できて調査効率が上がる
- ランタイム・依存関係は `package.json` / `requirements.txt` / `go.mod` / `Cargo.toml` / `pom.xml` などから抽出する
- 定数は `const [A-Z_]+\s*=` や `[A-Z_]+\s*=` のようなパターンを Grep すると漏れにくい
- 環境変数は `docker-compose.yml` や `.env.example` の列挙と、ソースコード中の `process.env.X` / `os.Getenv("X")` / `ENV["X"]` などの参照の両方を突き合わせる
- 同一プロジェクトに既存の Design Doc があれば、章立ての粒度・理由ラベルの書き方・表の作法の参考として最初に目を通すと、文体の一貫性が得やすい
